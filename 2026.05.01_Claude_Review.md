# InstallNewComputer — Claude Review (2026-05-01)

Two main programs: `scripts/setup_macos_apps.sh` (57 lines, bash bootstrap) and `scripts/setup_macos_apps.py` (313 lines, curses checklist UI driving Homebrew + `mas`).

## setup_macos_apps.sh

### Looks
- Plain terminal output. Appropriate for a bootstrap script — anything fancier would be wrong here.

### Behavior
- Sensible chain: detect admin, check Homebrew, fall back to installing it, then hand off to Python. The "must not run with sudo" guard is correct and well-documented in the README.
- Good that it uses `sudo -v` once before the non-interactive Homebrew installer rather than wrapping the whole script.

### Architecture
- 57 lines is the right size for the role. Don't grow it.

## setup_macos_apps.py

### Looks
- Curses checklist. Functional and dependency-free, which is the right trade for a "first run on a new machine" tool — no pip step needed.
- A cleaner two-column layout (formulae on left, casks on right) would scan better when the lists grow, but it's fine for now.

### Behavior
- Static lists (`BREW_FORMULAE`, `MAS_APPS`, `BREW_CASKS`) make the operative content trivially auditable — a strength. The dataclasses (`Formula`, `Cask`, `MasApp`) give type-safety with no overhead.
- Worth: when an item fails to install, accumulate failures and print a final summary at the end rather than letting them scroll past during the run.

### Architecture
- 313 lines is at the upper bound but still single-purpose. If it grows further, split: `apps_list.py` (data), `installer.py` (subprocess wrappers), `tui.py` (curses).
- Consider externalizing the lists to a TOML/YAML file so the user can edit without touching Python — but only if you ever want to share this with others.

## Cross-cutting recommendations
- Add a "dry-run" flag that prints the brew/mas commands without executing.
- Pin a tested `mas` version, since `mas-cli` has had reliability issues across macOS releases.
- A small `tests/test_lists_unique.py` to assert no dup app IDs would be cheap insurance.

## What NOT to change
- The bash-first entry point. It's the right call given the "fresh machine, no Python yet" scenario.
- Curses for the UI — adding any pip dependency here would defeat the purpose.
- The hard refusal to run under `sudo`.
